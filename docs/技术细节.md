## 名词解释

### 入口与流量路径

- **反向代理（Reverse Proxy）**：部署在用户与后端应用之间，负责 TLS、转发、限流与日志。
- **Nginx / OpenResty**：入口网关实现，OpenResty 适合承载可编排网关逻辑。

### WAF 与规则体系

- **WAF**：HTTP 层防护与审计系统。
- **OWASP CRS**：通用规则集，本身不是执行引擎。
- **WAF 引擎**：执行规则并输出审计日志（本项目使用 Coraza 兼容链路）。
- **SecLang**：规则语言。
- **DetectionOnly**：仅检测不阻断（当前默认策略）。
- **Block/Denied**：阻断模式，需结合阈值与审批策略。
- **Anomaly Scoring**：按规则命中累计异常分并触发动作。

### 处置与自动化

- **Rate Limit**：限流，低风险常见动作。
- **Block IP**：封禁，风险高于限流。
- **Challenge**：挑战动作（介于限流与封禁之间）。
- **TTL**：动作生存时间，到期自动失效。
- **Rollback**：撤销动作，闭环必须具备。

### 取证与证据

- **tshark**：命令行抓包工具。
- **pcap**：流量证据文件。
- **取证窗口**：围绕事件时间点抓取前后固定区间。

## 总体框架

### 数据存储层

#### 需要处理的数据类型（按重要性）

1. **入口层/WAF 日志**：Nginx/OpenResty 与 WAF 审计日志。
2. **结构化事件**：统一为 `SecurityEvent`（`ts/src_ip/uri/status/rule_id/rule_score/...`）。
3. **闭环记录**：`alert/incident/action/approval/audit`。
4. **取证材料**：`pcap` 与元数据。
5. **资产与策略配置**：资产、阈值、动作策略。

#### 存储组合（从 MVP 到增强）

**MVP（当前已落地）**

- **PostgreSQL**：结构化事实数据（事件、事件单、动作、审批、取证、审计）。
- **Redis（启用）**：动作状态缓存与限流计数状态。
- **本地文件系统**：日志归档、pcap、报告文件。

**增强（后续）**

- **ClickHouse**：高频日志/事件聚合查询。
- **对象存储（S3/OSS/MinIO）**：pcap 与报告归档。

### 业务逻辑层

#### 核心业务逻辑

**WAF/访问日志 -> 解析标准化 -> LLM 直分析 -> 风险分级 -> 低危自动处置/高危审批 -> 回写审计与可视化**

#### 主流程（当前实现导向）

1. **Ingest**：采集 Coraza JSON 审计日志。
2. **Normalize**：解析为统一 `SecurityEvent`。
3. **LLM Analyze**：调用 LLM API 生成 `Incident` 草案与处置建议。
4. **Policy + Approval**：低危自动动作，高危审批门控。
5. **Actuate**：动作入库并同步 Redis 状态键，支持回滚。
6. **Forensics**：满足条件时触发 tshark 抓包并记录元数据。
7. **Visualize**：仪表盘读取统计与工单时间线。

#### 逻辑子系统（MVP）

- 日志采集与解析（Ingestion/Normalization）
- LLM 分析与工单（LLM/Incident）
- 决策与审批（Policy/Approval）
- 执行与回滚（Actions/Actuator）
- 取证（Forensics）
- 表盘与账号（Dashboard/Auth/Settings）

#### LLM 调用保障（当前状态）

当前 `backend/src/services/llm/incident-analyzer.ts` 已切换为真实 LLM API 调用，会向 `LLM_API_URL` 发起请求，请求体包含运行时 `model`、`task`、本地渲染后的中文 `prompt`、`prompt_version`、`prompt_digest` 与事件输入，并通过 `AbortController` + `LLM_TIMEOUT_MS` 做超时控制。返回结果已使用 `llmIncidentAnalysisSchema` 与 `llmIncidentAnalysisEnvelopeSchema` 做 schema 校验，兼容直接结构与 `{ data: ... }` 包装结构。

当前 LLM 调用链路已落地以下治理能力：

- 本地 Prompt registry：`backend/src/services/llm/prompt-registry.ts` 负责按 `task + prompt_version` 渲染中文 Prompt，并计算 `prompt_digest`。
- 调用可靠性：已实现重试、退避、进程内熔断与确定性降级；当上游不可用或熔断打开时，可退化为基于事件规则的本地摘要分析。
- 调用审计：`POST /api/incidents/analyze-events` 与 MCP `analyze_incident` 都会记录 `provider/degraded/attempts/retries/latency/task/prompt_version/prompt_digest/model/model_version/input_digest`。
- 结果留痕：`llm_reports` 会记录 `model/task/prompt_version/prompt_digest/input_digest`，用于追溯当前事件单对应的模型版本和 Prompt 版本。

当前仍存在的边界：

- 熔断状态当前只保存在单进程内存中，多实例部署时不会共享。
- Prompt registry 目前是代码内静态注册，尚未提供管理接口、灰度切换或专门测试覆盖。
- `POST /api/incidents/:id/llm-reports` 手工写报告接口允许外部直接写入 `task/prompt_version/prompt_digest/input_digest`，但不会替代真实分析链路。

## 细节补充

### 功能单元与数据流转

单向数据管道（当前准则）：

`Logs -> Parser -> Event Store -> LLM API Analyze -> Incident Store -> Policy -> (Auto Action | Approval) -> Actuator -> Action Log + Redis Action State -> Dashboard`

关键约束：

- 任何执行必须写入 `actions` 表，且可回滚。
- 高危动作必须先进入 `approvals`。
- Redis 只保存动作状态缓存，事实数据仍以 PostgreSQL 为准。

### 数据库与缓存设计

#### PostgreSQL 核心表

- `assets`
- `events_raw`
- `alerts`
- `incidents`
- `actions`
- `approvals`
- `forensics`
- `policies`
- `llm_reports`
- `audit_logs`

说明：`llm_reports` 当前除分析结果外，还会记录 `model/task/prompt_version/prompt_digest/input_digest`。

#### Redis 键设计（当前已落地）

- 前缀：`REDIS_KEY_PREFIX`（默认 `waf:mvp`）
- 动作状态键：`<prefix>:active_action:<scope>:<action_type>:<target_base64url>`
- 值：`action_id/incident_id/requested_by/executed_by/cached_at/...`
- TTL：取动作 `ttl_seconds`

### 核心流程（按当前实现）

1. 审计日志命中后写入 `events_raw`。
2. 调用 `POST /api/incidents/analyze-events`，由 LLM 直分析生成事件单。
3. 生成 `incident/alert/llm_report/audit_log`，并同步写入 Prompt/模型版本元数据。
4. 执行动作时：写 `actions`（初始 `pending`）+ 写 Redis 动作状态，并异步触发网关验证收敛结果。
5. 回滚动作时：写回滚 `actions`（初始 `pending`）+ 删除 Redis 动作状态键，并异步验证恢复结果。
6. 异步收敛阶段：验证器重试探测网关状态并回写 `actions.result`，同时记录结构化 `action_receipts`；超时由 watchdog 兜底标记 `fail` 并审计。
7. 需要取证时：写 `forensics` 任务（`queued`），异步触发 worker 回写状态（`capturing/completed/failed`）。
8. 下载取证文件时：先通过 `GET /api/forensics/:fid` 获取签名下载 URL，再请求下载接口获取 pcap 文件流。

## 项目架构

- **入口防御**：Nginx/OpenResty + Coraza/ModSecurity + OWASP CRS
- **后端**：Node.js（主）+ Python（取证 worker 可选扩展）
- **存储**：PostgreSQL（事实数据）+ Redis（动作状态缓存）+ 本地文件（日志/pcap/报告）

- **表盘**：Web（Vue 3 + Vite 骨架）+ 后续可扩展实时推送
