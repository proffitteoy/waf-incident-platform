## 二、核心设计原则

### 1. 分层与边界

- 严格 `API -> Service -> Data`，不跨层调用。
- 事实数据以 PostgreSQL 为准，Redis 仅作缓存状态。

### 2. 配置优先

- 环境变量 > 配置文件 > 默认值。
- 基础服务（如 Redis）若为系统前置条件，启动阶段必须显式校验。

### 3. 可审计闭环

- 所有动作必须落库并可回滚。
- 高风险动作必须经过审批流。

### 4. 文档同步

- 架构变化必须同步更新 README 与架构文档。
- 目录级 README 必须包含：内容、边界、接口、关键函数/入口。

### 5. 可测试与可演进

- 核心链路可单测/集成测。
- 可扩展为异步 worker、实时推送、对象存储归档。

## 三、落地检查清单

1. 路由与数据流是否与架构图一致。
2. Redis/数据库健康检查是否可观测。
3. 动作执行与回滚是否双向闭环。
4. LLM 输出结构是否做 schema 校验。
5. 每个目录是否存在并维护 README。
